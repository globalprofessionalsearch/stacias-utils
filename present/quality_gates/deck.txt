WHAT HAPPENED?
==============

Couldn't get Sonar work the way we need
- Tried many things
- Ultimately run into too many constraints
- Project was taking too long
- Took a different direction

Had to consider our goals and redirect.

__NEW_SLIDE__

SONAR'S LIMITATIONS
===================

- Only one linter per repo
  - we have two (golangci-lint, spectral)
- Difficult integration of external linter with quality gates
  - Severity and priority levels do not map cleanly
- Hysteresis difficult to grasp, unwieldy
  - True for both lints and coverage
- Insufficient analysis for Go

Needed to fall back to what our primary needs are.

__NEW_SLIDE__

JEENIE'S NEEDS
==============

- We need to be in driver's seat
- Not limited to the scope and capabilities of a 3rd party tool.
- Not interested in feel-good charts and graphs.
- Need the build to enforce standards
- Need control over lint rules, and access to tools not available in Sonar

__NEW_SLIDE__

Travis
======

Runs the makefile commands as follows:
 - `make lint`
 - `make test-race`
 - `make enforce-code-standards`

Will fail the build if tests fail or if standards are not met.
 
__NEW_SLIDE__

Github Actions
==============

Github Actions only enforces static analysis standards (currently).

Doesn't have access to decryption keys for gcp config
Can overcome this limitation later.
 
 Runs the makefile commands as follows:
 - `make lint`
 - `make enforce-static-analysis-standard`

Will fail the build if tests fail or if standards are not met.

__NEW_SLIDE__

Local Builds
============

Engineers should install `golangci-lint `and `spectral` IDE extensions.
 
Should also run the following periodically during developement:
 - `make lint`
 - `make test-race`
 - `make enforce-code-standards`

How This Works - make lint
==========================
We have two linters for solo-platform-api

`spectral` 
 - analyzes oapi spec standards
 - configured via `.spectral.yml`

`golangci-lint` 
 - analyzes all go code 
 - configured via `.golangci.yml`
   
__NEW_SLIDE__

How This Works - make lint
==========================
   
New cli utility, located in `cmd/codebase-utils`

codebase-utils 
 - Contains tools that are specific to managing the codebase
 - Must never be used to manage the state of the platform
 - (that is what the `solo-platform-api` commands are for)
 - used to consolidate results from multiple linters
 - also other tools for coverage analysis etc...

We have colocated `solo-platform-api` and `codebase-utils` 
 - They are now both in the `cmd` directory
 - This is idiomatic for larger go project structures

 Standard defined by two metrics:
    Critical Line Count

__NEW_SLIDE__

How This Works - make enforce-code-standards
============================================
Critical Line Count

- Number of linter issues with a `critical` severity.
- Any count at or above this number fails the build
- Should be set to zero.
- Managed in makefile via `LINT_CRITICAL_COUNT_STANARD`

- spectral issues
    - always critical severity
    - regardless of .spectral configuration

- golangci-lint
    - Highly configurable rules and severities
    - Only small number of linters are set to critical
    - We will expand this over time
    - Most rules currently disabled

__NEW_SLIDE__

How This Works - make enforce-code-standards
============================================
Total Line Count

 - Total number of linter issues (regardless of severity)
 - Total count at or above this number fails the build
 - Managed in makefile via `LINT_TOTAL_COUNT_STANDARD`
 - We will reduce this number over time as we improve quality.
